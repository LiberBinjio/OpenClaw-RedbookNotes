### Reference Pattern B｜功能点升级为路径型
- 常见爆点：不是列一堆功能，而是给一条从输入到结果的可执行路径。
- 可借用模式：`功能介绍` 改写成 `步骤链路 + 场景结果 + 可复用动作顺序`。
- 本轮映射：把 Skill 写成 workflow 的一个节点，而不是单独炫功能。

### Reference Pattern C｜前后落差型
- 常见爆点：以前装了工具也不知道怎么用，现在内容开始直接告诉你怎么接入真实流程。
- 可借用模式：用“装了但没跑通”对比“接进流程后能稳定交付”。
- 本轮映射：适合创作者、开发者、内容团队共同代入。

### Reference Pattern D｜结果交付型
- 常见爆点：用户不关心技术名词，用户关心的是能不能更快完成一件具体任务。

## Viral Copy Breakdown
### 1. Hook Pattern
- 高传播内容常用结构：`你以为重点还在安装/工具名，其实真正拉开差距的已经是使用路径/交付结果`。
- 本轮保留：`安装教程` 对比 `工作流接法` 的认知反差。
- 本轮改写：把旧关注点写成“怎么装、怎么配、哪个模型更强”，把新关注点写成“装上后放在哪个节点、能不能少走几步、能不能稳定复用”。
- 本轮禁抄：任何现成爆文的反转标题原句。

### 2. Structure Pattern
- 推荐结构：
  1) 先指出旧写法还停在安装教程
  2) 再点出新的内容分水岭是“安装后如何跑通流程”
  3) 用 3-4 条 Newrank 邻近线索并列证明
  4) 最后给一个今天就能执行的内容改写动作
- Draft 可直接套成：反差开头 → 场景痛点 → OpenClaw/Skill/Newrank 公开线索 → 为什么路径型内容更容易被收藏 → 行动建议。
- 本轮禁抄：资讯原文段落顺序、导语句、总结金句。

### 3. Conflict Point
- 表层冲突：很多内容还在教“怎么装 OpenClaw/怎么接一个 Skill”。
- 深层冲突：更高阶的创作者和团队已经在讨论“装上之后怎样进入 workflow，最后怎样稳定交付”。
- 这个冲突值得保留，因为天然适合评论区讨论与身份代入。

### 4. Promise
- 给读者的承诺不该是“立刻爆”，而应是：知道下一阶段更该写什么内容、更该沉淀什么流程资产。
